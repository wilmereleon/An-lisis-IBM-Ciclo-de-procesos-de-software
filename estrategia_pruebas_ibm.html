🎯 Documento de Estrategia de Pruebas

IBM Quality Management System - Test Strategy Document

Conforme a IEEE 829, ISO/IEC 25010, TMMi Level 3, CMMi Level 3

📋 Resumen Ejecutivo

Este documento define la estrategia integral de pruebas para el IBM Quality Management System, estableciendo objetivos, alcance, enfoque, recursos y métricas de calidad alineados con los estándares internacionales de testing y calidad de software.

🎯Objetivos de Pruebas

Objetivo General

Garantizar la calidad, confiabilidad y rendimiento del IBM Quality Management System mediante un proceso de pruebas sistemático, automatizado y alineado con los estándares TMMi Level 3 y CMMi Level 3.

Objetivos Específicos

  • Calidad Funcional: Verificar que todas las funcionalidades cumplan con los requisitos especificados y expectativas de usuario
  • Rendimiento: Validar tiempos de respuesta < 2s para el 95% de las transacciones bajo carga normal (100 usuarios concurrentes)
  • Seguridad: Identificar y mitigar vulnerabilidades de seguridad siguiendo el OWASP Top 10
  • Compatibilidad: Asegurar funcionamiento correcto en múltiples navegadores (Chrome, Firefox, Safari, Edge) y dispositivos
  • Cobertura: Alcanzar mínimo 80% de cobertura de código con pruebas automatizadas
  • Regresión: Garantizar que nuevos cambios no introduzcan defectos en funcionalidades existentes

🧪Tipos de Pruebas

Pruebas Funcionales

Unit Testing

Validación de componentes individuales (funciones, métodos, clases)

Herramientas: Jest (frontend), Mocha/Chai (backend) | Cobertura Objetivo: >85%

Integration Testing

Validación de interacciones entre módulos y componentes

Herramientas: Jest, Supertest | Cobertura Objetivo: >75%

End-to-End Testing

Validación de flujos completos de usuario en entorno real

Herramientas: Cypress, Selenium | Cobertura Objetivo: Flujos críticos 100%

API Testing

Validación de endpoints REST, contratos, códigos de respuesta

Herramientas: Postman/Newman, SoapUI | Cobertura Objetivo: Todos los endpoints

Pruebas No Funcionales

Performance Testing

Validación de tiempos de respuesta, throughput y uso de recursos

Herramientas: JMeter, LoadRunner | Objetivo: <2s para 95% transacciones

Load Testing

Validación de comportamiento bajo carga esperada y picos

Escenarios: 100 usuarios (normal), 500 usuarios (pico), 1000 usuarios (stress)

Security Testing

Identificación de vulnerabilidades (OWASP Top 10)

Herramientas: OWASP ZAP, SonarQube | Objetivo: 0 vulnerabilidades críticas

Usability Testing

Validación de experiencia de usuario, accesibilidad (WCAG 2.1)

Herramientas: Lighthouse, axe DevTools | Objetivo: Score >90

✓Criterios de Entrada/Salida

Criterios de Entrada (Entry Criteria)

  • ✅ Código fuente disponible en repositorio Git con branch actualizada
  • ✅ Ambiente de pruebas configurado y operacional
  • ✅ Requisitos de usuario y especificaciones técnicas aprobadas
  • ✅ Casos de prueba diseñados, revisados y aprobados
  • ✅ Datos de prueba preparados y cargados
  • ✅ Herramientas de testing configuradas (Cypress, JMeter, Postman)
  • ✅ Build exitoso en CI/CD sin errores de compilación

Criterios de Salida (Exit Criteria)

  • ✅ Ejecución del 100% de los casos de prueba planificados
  • ✅ Cobertura de código ≥80% (unit tests)
  • ✅ 0 defectos críticos abiertos (severity 1)
  • ✅ ≤5 defectos mayores abiertos (severity 2), con workarounds documentados
  • ✅ Tasa de éxito de pruebas ≥95%
  • ✅ Performance dentro de SLAs establecidos (response time <2s)
  • ✅ 0 vulnerabilidades de seguridad críticas o altas
  • ✅ Aprobación formal de stakeholders (sign-off document)
  • ✅ Documentación completa (test reports, defect logs, metrics)

Criterios de Suspensión (Suspension Criteria)

  • 🛑 >20% de casos de prueba fallando repetidamente
  • 🛑 Defectos críticos que bloquean funcionalidades core
  • 🛑 Ambiente de pruebas inestable o no disponible >4 horas
  • 🛑 Cambios mayores en requisitos no aprobados formalmente
  • 🛑 Recursos críticos (personal, herramientas) no disponibles

Criterios de Reanudación (Resumption Criteria)

  • 🔄 Defectos bloqueantes corregidos y verificados
  • 🔄 Ambiente estabilizado y validado
  • 🔄 Nueva versión del software disponible con fixes
  • 🔄 Aprobación formal del Test Manager para reanudar

🐛Gestión de Defectos

Severidad Descripción SLA Resolución Ejemplo
CRÍTICA (S1) Sistema completamente inoperable, pérdida de datos, seguridad comprometida 24 horas Sistema de login no funciona, pérdida de base de datos
ALTA (S2) Funcionalidad crítica afectada, workaround complejo o inexistente 72 horas Reportes no se generan, métricas incorrectas
MEDIA (S3) Funcionalidad afectada, workaround disponible 1 semana Filtro de dashboard no funciona correctamente
BAJA (S4) Problema cosmético, impacto mínimo en funcionalidad 2 semanas Alineación de texto incorrecta, typo en label

📊 Proceso de Gestión de Defectos

1. Detección → 2. Registro (Jira) → 3. Triage (asignación severity/priority) → 4. Asignación (Dev Team) → 5. Corrección → 6. Verificación (QA) → 7. Cierre → 8. Lecciones Aprendidas
